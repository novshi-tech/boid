# バイブコーダー向け boid プロビジョニング 設計ドキュメント

ステータス: 設計中
作成日: 2026-06-10

---

## 背景と解決する課題

boid は Linux のローカル環境を活かす前提のオーケストレータであり、
素の状態では非エンジニアには到底セットアップできない。一方で「コードは書けないが
Claude Code はネイティブに使う」層 — いわゆるバイブコーダー — から利用要望がある。

この層の重要な性質は、**Claude Code（エージェント）が第一言語**である点。
あらゆる作業をエージェントに依頼するのが自然で、GUI のフォーム操作よりも
「エージェントに口で頼む」ほうが負担が小さいと感じる。

したがって本ドキュメントが解く課題は「非エンジニアでも boid を立ち上げ・運用・復旧できる
パッケージング」であり、その手段として **VM イメージを焼くことではなく、
冪等なセットアップ自動化（Infrastructure as Code）を整備すること** を中心に据える。

### 配布物は「VM イメージ」ではなく「セットアップ自動化」

当初は VMM イメージ（ova / qcow2）に固める案を検討したが、以下の理由で退けた。

- VM イメージは更新のたびに焼き直しが必要で、boid の「PR マージで自動更新」文化と噛み合わない。
- クラウド VPS に置く場合、配布用イメージは不要で「素の Ubuntu に
  セットアップスクリプトを流す」だけで済む。
- イメージは IaC スクリプトを「焼き固めた」副産物にすぎない。一次成果物はスクリプトであり、
  イメージ化（Packer 等）は配布形態として後から選べる。

この設計のもう一つの効能として、**この環境自体の障害復旧（DR）にも流用できる**。
手順を人間の記憶に置かず、新しい箱にスクリプトを流せば同じ環境が蘇る。

## 設計の背骨: エージェント・ファースト

本設計の全体を貫く原則。

- **人間の主インターフェースは Claude Code。** boid はそのエージェントが操作する対象。
- **WebUI はモニタリングと補助。** 主従はエージェントが主、WebUI が従。
- セットアップ・プロジェクト初期化・日常運用のすべてを「エージェントに頼む」形に寄せる。

この原則により、従来は別々だった論点 — セットアップ自動化、プロジェクト clone、
外部エージェント連携（MCP）— が一本の線上に並ぶ。

## ホスティング形態の選択

| 選択肢 | コスト感（16GB 目安） | 備考 |
|---|---|---|
| 大手クラウド（AWS / GCP） | 月 1〜2 万円 | 開発用メモリを積むと割高 |
| 欧州系 VPS（Hetzner 等） | 月 2 千円前後 | 最安。一人運用なら十分 |
| さくらの VPS | 月 1 万円前後（要確認） | 国内・円建て・日本語サポート |
| ローカル VM（Mac の Lima / UTM 等） | クラウド代ゼロ | スリープ問題あり |

補足:

- boid + Claude Code は常時フルメモリを消費しない。重いのはビルド・テスト・docker の
  瞬間だけでピークが短い。8〜16GB + 多めの swap で実用上しのげる。
- Web UI 越しの操作は非同期なのでレイテンシの影響は小さく、国内 VPS の優位性は限定的。
- コスト優先なら海外 VPS、安心・円建て優先ならさくら、という住み分け。

## セットアップ対象の抽象化

「SSH に限るか」ではなく「**対象を何と定義するか**」で考える。
対象を **「Ubuntu 24.04 LTS のシェルが一個ある」** という抽象まで持ち上げれば、
リモートでもローカルでも同一の本体スクリプトで構成できる。違うのは命令の届け方だけ。

### 三形態

| 形態 | 届け方 | boid 動作の確実性 |
|---|---|---|
| リモート VPS | SSH | 確実（標準 Ubuntu） |
| Mac ローカル Linux VM（Lima / UTM / OrbStack, arm64） | SSH | 確実（中身は標準 Ubuntu） |
| Windows WSL2 | `wsl` exec（or 内部 sshd） | **要検証**（後述） |

- Linux ネイティブのデスクトップは、本体スクリプトを直接叩くだけなので自明。後回しで可。
- **本体スクリプトは経路非依存に書く。** 届け方（SSH / wsl / ローカル直実行）は
  セットアップスキルのラッパー層が吸収する。本体は一個、届け方は複数。

### WSL2 のリスク（直感と逆）

ネイティブ統合の WSL2 のほうが、VM を噛ませる Mac より boid 的にはリスクが高い。

- WSL2 のカーネルは Microsoft のカスタムカーネルで、標準 Linux そのものではない。
- boid のサンドボックスは pasta（ネットワーク分離）+ nftables（egress フィルタ）+
  user namespaces という「標準 Linux のカーネル機能フルセット」を前提にする。
- user namespaces はおそらく動くが、**pasta / nftables が WSL2 のカスタムカーネル +
  NAT 環境で動くかは未検証**。机上では断定できない。
- 対して Mac の Lima 等は中身が本物の標準 Ubuntu カーネルなので、boid の前提が確実に揃う。

正式サポートの優先順位: **リモート VPS（第一） > Mac ローカル VM（第二） > WSL2（要検証）**。

## ブートストラップの解決

「セットアップをエージェントにやらせるには、先にエージェントが動く環境が要る」という
鶏卵問題は、**ローカルの Claude Code を制御マシン（control node）にする**ことで解消する。

- バイブコーダーのローカルマシンには既に Claude Code がある（それが第一言語だから）。
- そのローカル Claude Code が SSH 越しに VM を外からセットアップする。
  VM 側には事前インストール不要。SSH で到達できればよい。
- Claude Code は内部で bash を使うため、「ローカルに Claude Code がある = bash 実行能力がある」は
  自動的に成立する。WSL2 / git bash の前提とも一致。

### 人間に残る最小手作業

完全ゼロタッチにはならない。VPS の VM 作成自体の自動化は別レイヤーとし、入口は以下に絞る。

1. VM を作る（VPS 管理画面 or ローカル VM ツール）
2. SSH 鍵を登録する
3. ローカルの Claude Code に IP / 鍵を渡して「boid を入れて」と頼む

### セットアップと日常運用の二層

- **セットアップ**はローカル Claude Code から（ブートストラップ回避）。
- **日常運用**は、セットアップ時に VM 側にも Claude Code を入れておき、そちらで操作するか、
  WebUI で行う。セットアップスクリプトに「VM 側 Claude Code のインストール」を含めることで
  二層がきれいに分離する。

## ディストリ固定: Ubuntu 24.04 LTS

ディストリは一個に固定する。これは好みではなく必須要件。

### 根拠

- boid のサンドボックスは独立 rootfs を持たず、**ホストの `/usr` `/bin` `/lib` 等を
  読み取り専用 bind mount で共有する**（`internal/sandbox/plan.go:35-44`）。
  サンドボックス内バイナリはホストの共有ライブラリ（glibc 等）にリンクして動くため、
  **ホストのディストリ = 実行環境そのもの**。バラつくと即破綻する。
- サンドボックスが依存する `pasta`（passt パッケージ）が apt で素直に入る世代である必要があり、
  22.04 では古い。24.04 が妥当。

### 注意点: unprivileged user namespaces の制限

boid は `unshare --user`（`internal/sandbox/render.go:56`）で unprivileged user namespace を
使うが、近年の Ubuntu（24.04 含む）は AppArmor によりこれをデフォルトで制限している。
素の 24.04 ではサンドボックスが起動しない可能性があり、**緩める設定（sysctl or AppArmor
プロファイル）がセットアップに必要**。正確な効かせ方は実機で詰める（宿題）。
この「ディストリによって挙動が違う」こと自体が、固定して一個の正解を握るべき理由。

## ツールのプロビジョニング: ホストに全部盛り

### 根拠: サンドボックスの構造が方針を決める

- サンドボックスはホストの `/usr` を読み取り専用共有し、**内部に apt / pip を持たない**。
  必要なツールはすべてホストから bind mount で差し込む設計
  （`internal/dispatcher/sandbox_builder.go` 周辺、git は shim 経由で `/usr/bin/git` に bind mount）。
- したがって **「サンドボックスで使えるツール = ホストに入っているツール」**。
  Claude にサンドボックス内でツールを追加させるのは構造的に不可能（`/usr` は ro、apt も無い）。
- 帰結: **ツールはホストに pre-install する一択**。プロビジョニング対象は「ホスト一個」で済み、
  サンドボックスごとの環境作り分けは不要。

### 方針

- 非エンジニアは後からツールを足せないので、**メジャーな言語ランタイム一式をホストに全部盛り**。
- リテラシーのある人は VM なので中を直接いじってカスタマイズできる。
  バイブコーダーは全部盛りの初期状態で満足して使う。kit に凝った仕掛けを仕込む必要はない。

### 基盤層（boid が動く最低ライン）

`passt`(pasta) / `nftables` / `util-linux`(unshare) / `git` / `docker` / `Go` / `Claude Code`
＋ user namespaces 有効化設定。

ユーザ層（全部盛りに含める言語ランタイムの線引き）は要決定（宿題）。

## セットアップ実装手段: bash + セットアップスキル

### Ansible ではなく bash シェルスクリプト

- Ansible は「人間が外から流す」前提でエージェントから見ると間接的。エラー時にモジュール知識が要る。
- bash の `install.sh` のほうがエージェントにとって扱いやすい（出力をそのまま読め、コケた箇所を
  見て直せ、状態を確認しながら進められる）。
- Windows の git bash でも動く。重い作業は VM 側の本物の Ubuntu bash で走り、git bash が担うのは
  「SSH して VM 側スクリプトを叩く」薄い層だけなので、パス変換等の癖を踏むリスクは小さい。
  git for Windows には ssh / scp も同梱。
- POSIX 寄りのシェルスクリプトで書けば Mac / Linux / Windows(git bash) を一枚でカバーできる。

### 再現性のガードレール

エージェントにフル即興させると環境がばらつき、IaC の再現性が崩れる。これを防ぐため:

- **スキルは即興しない。** 決定論的な `install.sh` を本体に持つ。
- セットアップスキルはその install.sh を実行する前後の **環境チェックとトラブルシュートを
  ナビゲートする**ラッパー。エージェントは環境差の吸収のみを担い、手順の骨はスクリプトが固定する。

## プロジェクトの clone / 初期化

従来コマンドラインでちまちまやっていた操作を、エージェント・ファーストで置き換える。

- **主軸: エージェント経由。** 「このリポジトリで boid プロジェクトを作って」と頼むと、
  ホスト側 daemon が clone + `project.yaml` 初期化を行う。サンドボックス内 git は broker 経由だが、
  clone 自体はホスト側 daemon の仕事なので整合する。
- **補助: WebUI から git URL 指定フォーム。** エージェントを使わず GUI でやりたい人向け。
  既存のプロジェクト作成フローの延長で実装可能。
- private リポジトリは認証の口（GitHub token 等）が必要。public なら不要。

### 関連アイデア（別スコープ・温める）

HTTP エンドポイントに MCP を実装し、**外部エージェントから boid を操作させる**。
手元の Claude Code から「このプロジェクトを boid に送って回して」といった連携が可能になる。
エージェント・ファーストの背骨上にある拡張だが、対象がパワーユーザ寄りなので本スコープとは分ける。

## バックアップ / 復旧

### 原則: スクリプトで再現できない種だけ

バックアップは「全状態の丸ごとスナップショット」ではなく、
**「セットアップスクリプトでも git リモートからでも蘇らないものだけ」** を取る。
IaC 思想の自然な帰結。

| データ | 復旧の手段 | バックアップ |
|---|---|---|
| OS・ツール・boid 本体 | セットアップスクリプトで再現 | 不要 |
| プロジェクトのソース | git リモートから再 clone | 不要（URL だけ要る） |
| **プロジェクト定義**（リモート URL, ブランチ, project.yaml 等） | エクスポートから再構成 | **必要** |
| 未コミットの変更 | 失う（こまめ push 前提） | 要判断 |
| タスク | 捨てる | 不要 |
| ペアリング / web_secret | 再ペアリング | 不要 |
| runtimes / セッションログ | 揮発 | 不要 |
| **Cloudflare tunnel 設定** | 再設定が面倒 | **必要**（別カテゴリ・秘密） |

### 推し実装: プロジェクト定義のエクスポート / インポート

DB 丸ごとではなく `boid project export` / `import` で宣言的にエクスポートする。

- DB 丸ごとは不要なタスク・ペアリングを巻き込み、バイナリで差分も追えない。
- 宣言的テキストなら「設定の version 管理」になり、git に置け、差分が見える。
- バックアップだけでなく **新マシンへの引っ越し・複数マシンの構成同期**にも効く。
- セットアップスクリプト（環境を蘇らせる）+ プロジェクト定義（プロジェクトを蘇らせる）の
  二本柱で、「新環境にスクリプト → 定義 import → git から再 clone → 復活」が一本道になる。

### Cloudflare tunnel（別カテゴリ）

boid の外側だが実用上一緒に扱いたい。ただし **秘密情報**。
tunnel credentials（UUID ごとの鍵）+ config.yml を取れば別マシンで同じ tunnel を蘇らせられるが、
鍵を平文で git に置けない。**「プロジェクト定義（平文 OK）」と「シークレット類（暗号化必須）」の
二系統に分けて**扱う。
