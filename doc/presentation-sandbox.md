## 1. 背景

### AI エージェントに任意スクリプト実行を任せる時代

- Claude Code のような AI コーディングエージェントは、自然言語指示に応じて **コマンドを実行し、ファイルを書き換え、git を操作する**
- 便利だが、与えられた権限がそのままホスト OS の権限になる
- 誤爆・プロンプトインジェクション・不完全な指示 → **ホスト側に副作用が漏れる**

### 求められるもの

> 「エージェントが自由に動ける作業場」と「ホストの保全」の両立

→ 軽量で起動が速く、権限を細かく絞れる **サンドボックス** が必要

---

## 2. boid とは

**「汎用パーソナル AI オーケストレータ」**

- Linux 上でタスクを管理し、状態遷移に合わせて AI エージェントやスクリプトを発火させる常駐型の仕組み
- 「タスクがどの状態にあるか」「何が起きたら何が動くか」を宣言的に定義
- AI エージェントへの作業依頼は **必ずサンドボックス越し**

### 本発表のスコープ

boid の機能のうち、**「エージェントを安全に走らせる仕組み（サンドボックス層）」** に絞って解説します。

---

## boid の登場人物（概念レベル）

| 概念 | 役割 |
|---|---|
| **Task** | 作業単位。状態（pending / executing / verifying / done …）を持つ |
| **Hook** | 状態遷移に紐づき、実際の作業を行うスクリプト |
| **Gate** | 状態の入場/退場時に走る検証ロジック。合否を判定 |
| **Payload** | Task に添付された実行データ。Hook/Gate に渡される入力 |
| **Role** | 実行コンテキストの種別（hook / gate / command） |
| **Sandbox** | Hook/Gate/Command を隔離して実行する環境 |
| **Broker** | サンドボックス内から外界への唯一の通信窓口 |

このうち後半 3 つが本発表の主題です。

---

## 3. 脅威モデル

### 「サンドボックス内のプロセスはホストから信頼できない」

想定する具体的な脅威:

- 誤った `rm -rf ~` でホームディレクトリが消える
- `.git/config` 改ざんによる認証情報の窃取
- 許可されていないリモートへの `git push`
- `ssh` / `curl` によるクレデンシャル流出
- fork bomb や無限ループによるリソース枯渇
- 他プロジェクトのファイルを書き換える越境

**前提**: エージェントは善意でも、LLM の誤出力は普通に起きる。
**方針**: **拒否をデフォルト** に、必要な権限だけを明示的に付与する。

## 4. 構成の全体像 — スコープ 3 段 × 系統 3 種

サンドボックスの構成は単一ではない。**いつ決まるか** で 3 層に分かれる。

|              | **System**<br>(常駐で 1 つ) | **Project**<br>(プロジェクト設定で定義) | **Job**<br>(Role × State で絞る) |
|--------------|------------------|--------------------------|-----------------------|
| **Network**  | proxy allowlist<br>nftables / 仮想 NIC 構成 | — | Role 非依存 |
| **Filesystem** | —             | 公開ディレクトリ集合     | hook×executing: 全公開 rw<br>hook×verifying: 全公開 ro<br>gate: 全非公開 |
| **Broker**   | —                | host command 群<br>builtin-command 群<br>(代表例: git) | hook: host command 不可<br>gate: host command 可<br>builtin は両 Role 可<br>(hook は git fetch/push 禁止) |

---

## 3 軸モデルの読み方

- 横軸 = **いつ決まるか**
  - **System**: boid 常駐プロセスと共に 1 つだけ存在
  - **Project**: プロジェクトの設定で宣言・収集される「素材」
  - **Job**: ジョブ起動時に Role と State を見てトークンに固定される
- 縦軸 = **何を統制するか** — ネットワーク境界 / ファイル可視性 / プロセス実行経路
- Job 層は **0/1 スイッチ**: Role ごとに「全公開 or 全非公開」「全許可 or 全不許可」
  Project で収集した集合を「どれを使うか」細粒度に選別する機能は**ない**
- 全体は **Linux の unshare / mount / nftables / pasta** の上に乗る

→ §5 は **System 層**、§6〜§7 は **Project 層**、§8〜§9 で **Job 層（Role + State）の絞り** を扱う

## 5. プロセス・ネットワーク隔離

※ 本章で扱うのは §4 表の **System 層における Network 構成**（proxy allowlist / nftables / 仮想 NIC はすべて常駐で 1 つ）。

### 使っている Linux の仕組み（用語解説）

- **Namespace（名前空間）**: カーネルが提供する「リソースのビュー分離」機能。mount / pid / net / user など種類別に存在
- **`unshare`**: 現在のプロセスから新しい namespace を切り出す syscall / コマンド
- **`chroot` 相当のルート変更**: プロセスが「見る」 `/` の位置を差し替え、外側のファイルを不可視にする
- **UID mapping**: ホストの一般ユーザ UID を、namespace 内では「root（UID 0）」に見せかける仕組み
- **`pasta`**: ユーザ空間で動く軽量な仮想ネットワークツール。KVM / Podman などでも使われる
- **`nftables`**: Linux のパケットフィルタ。iptables の後継

---

## 隔離の全体像

サンドボックスの起動時、概念的には次の順で「殻」を重ねていきます。

1. **ネットワークの殻** — pasta が仮想 NIC を生やし、サンドボックス内は独立したプライベートネットに閉じ込められる
2. **マウントの殻** — mount namespace を新しく作り、以降のマウント操作がホストに影響しないようにする
3. **ファイルシステムの殻** — tmpfs で新しいルートを作り、必要最低限のディレクトリだけを bind mount で「並べる」
4. **ユーザの殻** — UID mapping で「中では root、外では一般ユーザ」という状態を作る
5. **最後にスクリプト実行** — この殻の中で Hook / Gate が走る

---

## ネットワーク隔離の考え方

- サンドボックス内は **独立したプライベートネットワーク**（ホストから見ると仮想 NIC の先）
- nftables は **デフォルト拒否 + ループバック / ゲートウェイ / DNS のみ許可**
- 任意のリモートへ直接 TCP を張ることはできない

### では完全にオフラインか？ → No

- ホスト側で **組み込み HTTP(S) プロキシ** が起動し、サンドボックスの `HTTP_PROXY` / `HTTPS_PROXY` に注入される
- プロキシは **許可ドメインのリスト（allowlist）** を持ち、一致するホスト名にのみ通信を中継する
  - CONNECT 先ホスト名 / HTTP `Host` ヘッダを検査
  - 許可外のドメインは接続段階で拒否
- 例えば `api.github.com` は許可、`random.example.com` は拒否、という粒度で制御する

→ **外に出る経路はプロキシに一本化**、そこに allowlist が立っている構図

## ネットワーク構成図

```
  ┌──────────────── Sandbox (netns) ────────────────┐
  │                                                  │
  │   エージェント / shim                            │
  │     │                                            │
  │     │ ① HTTP(S)      ② Broker IPC               │
  │     │ (HTTPS_PROXY)   (UNIX socket / FD)         │
  │     ▼                 │                          │
  │   pasta 仮想 NIC      │                          │
  │     │                 │                          │
  │  ┌──┴─── nftables ────┴── default DROP ──┐       │
  │  │ allow: loopback / gw(10.0.2.2) / dns  │       │
  │  └───────────────────────────────────────┘       │
  └──────────────┬──────────────────┬────────────────┘
                 │ ① TCP to gw      │ ② socket
                 ▼                  ▼
        ┌────────────────┐   ┌─────────────────────┐
        │  HTTP Proxy    │   │  Broker             │
        │  (host 側)     │   │  ├─ token check     │
        │  allowlist:    │   │  ├─ builtin-command │
        │   api.github…  │   │  │   (git push 等)  │
        │   …            │   │  └─ host command    │
        └───────┬────────┘   │      (gh 等)        │
                │            └──────────┬──────────┘
                │ 許可ドメインのみ      │ 許可コマンドのみ
                ▼                       ▼
        ┌──────────────────────────────────────┐
        │         Internet / Host OS           │
        └──────────────────────────────────────┘
```

サンドボックスから外に出る経路は **2 本だけ**: ① プロキシ（HTTP(S) + allowlist）、② Broker（プロセス実行 + ポリシー）。
nftables はこの 2 本に収まらない通信を **IP レベルで切り落とす最後の砦**。

（**Broker の内部構造は次章 §6 で扱います**。ここでは「外への出口は 2 本」という像だけ押さえれば十分）

## 6. Broker — 外界とつなぐもう 1 本の経路

※ 本章で扱うのは §4 表の **Project 層における Broker の素材定義**（host command 群 / builtin-command 群）。Job 層での Role 絞りは §8 で扱う。

### なぜ必要か

- HTTP プロキシの allowlist は **System 層のグローバル設定** — 全ジョブ共通で、むやみに広げられない
- しかし実際には、ジョブごとに必要な外部通信先は異なる
  - あるジョブでは `git push` で特定リポジトリにだけ通信したい
  - 別のジョブでは `gh pr create` で GitHub API を叩きたい
  - ホスト側 boid への「タスク完了」報告、など
- これらを全部 System の allowlist に積むと、**関係ないジョブにも広い権限が漏れる**

→ **Broker** は、System allowlist を最小に保ったまま、**ジョブ単位で必要な外部通信だけをピンポイントに許す** ための仕組み
→ §5 のプロキシとは別経路で、**「このジョブだけに許された窓口」** を提供する

---

## Broker の考え方

```
  [Sandbox 内]                       [Host 側]

   shim コマンド
   （git / gh / boid …）
       │ リクエスト（JSON）
       ▼
   ─── 専用ソケット ───▶  Broker
                          ├─ トークン確認
                          ├─ ディスパッチ
                          │    ├─ builtin-command
                          │    └─ host command
                          └─ 結果を返す
       ◀──────────────────
   stdout / exit code
```

通信路は **サンドボックス内から外への 1 本のソケット** のみ。この上に、後述の 2 系統のディスパッチが乗る。

---

## Broker の中心設計 — トークンに紐づいた実行可能コマンドのホワイトリスト

Broker の安全性は、ほぼこの 1 点に集約される。

- ジョブを起動する時点で、Broker に **「このジョブで呼んで良いコマンドの集合」** を登録する
- 登録の結果返ってくる **短寿命トークン** がサンドボックスの env に注入される
- サンドボックス内からのリクエストは必ずこのトークンを添える
- Broker はトークンを引いて **対応するホワイトリスト** を参照し、載っていないコマンドは即拒否

### 性質

- **ホワイトリストはトークン（＝ジョブ）ごと** — 他ジョブのトークンでは何ひとつ動かせない
- **登録後は動的に広がらない** — 実行中にコマンドが追加されることはない
- **ジョブ終了と同時にトークン失効** — 漏洩の影響範囲を時間的にも封じる
- 併せて **登録時スナップショット**（git remote URL など）も固定され、後から差し替えられない

→ 「何が許されるか」は **ジョブ開始の一点** で確定し、以降は読むだけ

---

## Broker の上に乗る 2 系統

ホワイトリストに載った各コマンドは、実体として次のどちらかで処理される。

### ① builtin-command — Broker 自身が処理する組み込み操作

- 例: `git push` / `git fetch`、boid へのタスク状態報告
- **ホスト側プロセスを起動せず**、Broker の Go コード内で直接処理
- 登録時スナップショット（remote URL など）と突き合わせて強く制約できる
- 「boid が特別扱いしたい操作」はここに入る
- **代表例は git** — 機能が多く副作用も幅広いため、細かく制御できるよう builtin に取り込んでいる

### ② host command — ホスト側プロセスを起動する汎用経路

- 例: `gh pr create` ほか外部 CLI
- Broker がホワイトリスト照合 → **ホスト側で別プロセスとして起動**
- 新しい外部ツールを通したいときは、登録エントリを足せば済む

※ 「ホワイトリストに **何を載せるか / どう細かく絞り込むか**」は **§8 ジョブ状態と Broker 権限** で扱う

## shim とは何か

**shim（シム）**: 薄いラッパープログラム。本来のコマンドと同じ名前で置かれ、呼び出しを横取りして別の処理に流す。

サンドボックス内では:

- `git` を叩くと、実は boid の shim が動く
- shim はコマンド引数を検査し、
  - 安全なもの（`git status`, `git diff` など）は **そのままローカル実行**
  - 副作用のあるもの（`git push`, `git fetch` など）は **Broker に問い合わせる**

→ エージェントからは普通の `git` コマンドに見えるが、裏で関所を通っている

## 7. ファイルシステムの最小公開戦略

※ 本章で扱うのは §4 表の **Project 層における公開ディレクトリ集合の定義**。Job 層での Role × State 絞りは §8 / §9 で扱う。

### 使っている仕組み（用語解説）

- **bind mount**: あるディレクトリを別の場所から「見える」ようにする仕組み。コピーではなく、同じ実体を別パスで参照
- **read-only remount**: 一度マウントしたものを「書き込み不可」に切り替える
- **tmpfs**: メモリ上のファイルシステム。プロセス終了で消える
- **overlay**: 下層 RO + 上層 RW を重ねる仕組み（今回は直接は使わない）

---

## サンドボックス内から見えるファイルシステム

典型的な構成:

| 場所 | 公開範囲 | 書込可否 | ねらい |
|---|---|---|---|
| `/bin`, `/usr`, `/etc` 等 | ホストから bind | **読取のみ** | システムツール利用 |
| `/dev`, `/proc` | ホストから bind | 通常通り | 最低限のカーネル情報 |
| `/tmp` | tmpfs（新規） | 書込可 | ホストに影響しない一時領域 |
| 作業対象ディレクトリ | bind | 役割依存 | Hook はここを書き換える |
| 他プロジェクト | bind | **読取のみ** | 参照可、改変不可 |
| それ以外 | **公開しない** | — | 見えないものは壊せない |

## 8. ジョブ状態と Broker 権限

§9 が **ファイルシステム側**、§8 が **プロセス側（host command / builtin-command）** の話。
いずれも軸は **Role × State**。

| Role × State | host command | builtin-command |
|---|---|---|
| **hook × executing** | 原則不可 | 可（ただし `git push` / `git fetch` は禁止） |
| **hook × verifying** | 原則不可 | 可（読み取り系のみ。書き込みは §9 の FS ro で自然に封じられる）|
| **gate × \*** | 許可されたものは可 | 可 |

- **hook** はプロセス系を絞る — **内側でコードをいじる役**
- **gate** は host command が開く — **判定結果を外に伝える役**
- **command** は従来型の汎用実行（ポリシー次第）

→ §9 と並べて読むと:
- hook = **FS rw / 外への発信は絞る**（内作業）
- gate = **FS 不可視 / 外への発信が開く**（外発信）

---

## ポリシー — 「何ができるか」の宣言

§6 で Broker が引く **ホワイトリストの中身** は、ここで Role 単位に宣言される。

Role ごとに **許可コマンドとその使い方** をあらかじめ登録:

- どのコマンドを呼べるか（例: `gh`, `git`, `boid`）
- 許可されるサブコマンドの一覧（例: `gh` なら `pr` のみ）
- 許可/拒否されるオプション引数のパターン（glob）
- 許可される通信先（git の remote URL スナップショット等）

### 2 段階の絞り込み

1. Role で「大きなカテゴリ」を決める
2. ポリシーで「その Role 内で何が許されるか」を決める

→ 過剰な権限を段階的に削ぎ落とす

---

## git のポリシー — builtin だからこそ細粒度に守れる

git は多くのサブコマンドを持ち、副作用の幅も広い。
そのため Broker の builtin-command として取り込み、以下のような **細かい制御** を効かせている:

- **Role による切り分け** — hook からは `push` / `fetch` を全面禁止
- **危険な refspec** は shim 段階で弾く（ブランチ削除系など）
- **remote の向け先はジョブ登録時に固定** — 実行中の差し替えを許さない

詳細な個別ルールは実装に委譲し、ここでは **「細粒度に守れる土台が用意されている」** 点だけ押さえる。

---

## 9. ジョブ状態と書込権限

§8 の Role に加えて、**hook を発火させるタスク状態** によって FS の書込可否が切り替わる。

| Role × State | 作業ディレクトリ | ねらい |
|---|---|---|
| **hook × executing** | **書込可（rw）** | エージェントが実装や修正を行う「作業フェーズ」 |
| **hook × verifying** | **読取専用（ro）** | テスト・Lint 等で検証する「検証フェーズ」 |
| **gate × \***        | そもそも不可視 | 判定専門なので書込も読取も対象外 |

### なぜ分けるのか

- verifying で書込を許すと、**検証ロジックが対象を変更してしまう事故** が起こりうる
- 「見て判断するだけ」と「直す」を物理的に切り分けることで、**副作用の発生点を限定**
- 失敗した検証は `reworking` に戻され、再び executing で書ける状態になる（設計上の巻き戻し）

→ **Role = 何にアクセスできるか** / **State = どう扱えるか**、2 つの軸で最小権限を構成

## まとめ — boid サンドボックスが目指したもの

### 軽量サンドボックス × ワークフロー連動の権限設計

- **ローカルのファイルとツールをそのまま活かす軽量サンドボックス**
  - Linux namespace / mount / nftables / pasta を薄く重ねるだけ、秒で起動
  - コンテナイメージ不要、ホストの開発ツールチェインをそのまま利用可能
- **プロジェクト × ワークフローの状態に合わせて制約を最適化**
  - System / Project / Job の 3 層で「いつ決めるか」を分離
  - Role × State で hook/gate、executing/verifying ごとに FS 可視性・コマンド許可を切り替える

### Claude Code on the web / Codex web 版との違い

クラウド側サンドボックスは **ローカル資産から切り離される** が、boid は手元の環境を前提に **限定的かつ明示的に** ツールへの窓口を開く:

- **統合テスト / E2E テスト** をエージェント作業の一環として実行できる
- **タスク管理システム（Jira、GitHub Issues 等）** と緊密に連携
- **GitHub (gh / git) との PR ワークフロー** を Broker 経由で安全に自動化
- ホストの機密を守りつつ、**「開発者の手元で動く AI エージェント」の基盤**として機能する
